STEP 6 — Parser fix: title should not repeat series/description
---------------------------------------------------------------
User noticed series shows both in title and in its own field on Kindle cards.
Old parser's cleanTitle deliberately kept the trailing "(Series Book N)" paren
after stripping the post-colon description. Now that series is a first-class
display field, that's redundant.

Fix applied to both parsers identically:
    function cleanTitle(rawTitle, series) {
      let cleaned = rawTitle;
      if (series) cleaned = cleaned.replace(/\s*\([^()]*\)\s*$/, "");
      const colonIdx = cleaned.indexOf(":");
      if (colonIdx !== -1) cleaned = cleaned.substring(0, colonIdx);
      return cleaned.trim();
    }

Result:
    "Beasts in the Garden: A sci-fi novel (Convergence Book 1)"
      title       → "Beasts in the Garden"
      description → "A sci-fi novel"
      series      → "Convergence Book 1"

sortableTitle is unchanged (keeps the series paren for in-series sort grouping).

Updated test expectations in src/lib/parsing/__tests__/kindle-parser.test.ts
(5 assertions) and audible-parser.test.ts (1 assertion). All 49 tests pass.

BACKFILL MIGRATION
------------------
src/migrations/002_clean_titles.sql strips the trailing series paren from
existing rows:

    UPDATE books
    SET title = regexp_replace(title, '\s*\([^()]+\)\s*$', '')
    WHERE series IS NOT NULL
      AND title ~ '\([^()]+\)\s*$';

Apply with: psql "$DATABASE_URL" -f src/migrations/002_clean_titles.sql

Caveat: uq_books_duplicate includes title. If two rows differ only in their
series paren (same type/title-stem/authors/owner/purchase_date), the UPDATE
will fail with a unique violation. Would need manual resolution if that happens.

STEP 8 — ASIN as join key (design)
----------------------------------
ASIN = Amazon's 10-char product id. Embedded in the library DOM as:
  • Amazon mycd:   content-image-{ASIN} / content-title-{ASIN} on container divs
  • audible.com:   /pd/*-Audiobook/{ASIN} in cover/title anchors

Column decision: books.asin TEXT, nullable, NO unique constraint.
  • Shared books legitimately share an ASIN across both owner rows
    (tverkon's row + dverkon's row of the same audiobook).
  • Manually-entered books have no ASIN.
  • Plain-text Kindle imports leave it null.
Enrichment UPDATE filters book_type = 'AUDIBLE' AND asin = $N (owner-agnostic
so both shared rows update in a single statement).

KEY DECISIONS
-------------
- asin is nullable, no unique constraint (shared books share ASIN by design).
- Enrichment is ASIN-only and owner-agnostic — shared rows get the same
  enrichment via a single UPDATE.
- Blurb-wins: non-null enrichment overwrites; null enrichment preserves.
- K-Scrape emits a hybrid stub so one bookmarklet covers both Amazon tabs;
  audible.com needs its own scraper (A-Scrape) because the DOM is different
  and it serves a different purpose (enrichment, not import).
- Both bookmarklets use IndexedDB (klScrape vs aScrape) so they coexist.
- Separated enrich flow as server actions (not an API route) to mirror the
  CRUD pattern; node-html-parser must run server-side anyway.
